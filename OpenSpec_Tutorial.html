一、什麼是 OpenSpec？

OpenSpec 是一個規格驅動開發 (Spec-Driven Development) 框架。簡單來說，它幫你在寫程式之前，先用結構化的文件描述「要做什麼」、「怎麼做」、「做哪些步驟」，然後讓 AI 助手（如 Claude Code、Cursor、Copilot 等）按照這些規格去實作。

核心理念
  • 流暢而非死板 — 沒有強制的階段門檻，按邏輯順序做事
  • 迭代而非瀑布 — 需求會變，理解會深化，允許隨時調整
  • 簡單而非複雜 — 最小化設定負擔，快速開始工作
  • 為既有專案設計 — 用差異化修改，不需要從零開始
先想清楚 → 再動手寫 → 寫完歸檔

二、安裝前的準備

2-1. 安裝 Node.js

OpenSpec 需要 Node.js 20.19.0 以上版本。

1
打開瀏覽器，前往 https://nodejs.org
2
下載 LTS（長期支援）版本（頁面上較大的綠色按鈕）
3
執行下載的安裝檔，全部按「Next」→「Install」→「Finish」
4
驗證安裝成功，打開終端機輸入： 
node --version
應該會顯示 v20.x.x 或更高的版本號。

2-2. 確認你有 AI 編程工具

OpenSpec 支援 24 種 AI 工具，常見的包括：

  • Claude Code（Anthropic 的 CLI 工具）
  • Cursor
  • GitHub Copilot
  • Windsurf
  • Cline、RooCode、Gemini CLI 等

你至少需要安裝其中一個才能使用 slash commands。

三、安裝 OpenSpec

打開終端機，執行以下指令：

npm install -g @fission-ai/openspec@latest

這會把 OpenSpec 安裝為全域指令，之後可以在任何地方使用 openspec 命令。

什麼是「全域安裝」？

加上 -g 代表安裝到整台電腦，不只限於某個專案。安裝完後，你在任何資料夾都能執行 openspec 指令。

四、在專案中初始化 OpenSpec

切換到你的專案資料夾，然後執行初始化：

# 切換到你的專案
cd 你的專案資料夾路徑

# 初始化 OpenSpec
openspec init

初始化過程會：

  1. 詢問你使用哪些 AI 工具（例如選 claude）
  2. 選擇工作流程模式（預設為 core 核心模式）
  3. 自動建立所需的資料夾與設定檔

非互動式安裝（直接指定選項）

openspec init --tools claude            # 只設定 Claude Code
openspec init --tools claude,cursor     # 設定多個工具
openspec init --tools all               # 設定全部支援的工具
openspec init --profile core            # 指定使用核心模式

初始化後的資料夾結構

你的專案/
├── openspec/
│   ├── specs/        ← 規格文件（描述系統現在的行為）
│   ├── changes/      ← 變更文件（每個新功能/修改一個資料夾）
│   │   └── archive/ ← 已完成的變更歸檔
│   └── config.yaml  ← 專案設定（選用）

五、核心概念（先懂再用）

5-1. Specs（規格）

描述系統目前的行為，是「事實的唯一來源 (Source of Truth)」。規格使用 RFC 2119 關鍵字：

  • MUST / SHALL — 必須做到的行為
  • SHOULD — 建議但允許例外
  • MAY — 選用功能
範例

「使用者登入後，系統 MUST 在 30 秒內回傳 token。」

5-2. Changes（變更）

每個新功能或修改都是一個「Change」，包含一個專屬資料夾，裡面有 4 個文件：

文件用途
proposal.md提案 — 為什麼要做？範圍多大？高層級方向
specs/差異規格 (Delta Specs) — 要新增/修改/刪除哪些行為
design.md設計 — 技術架構、實作決策（可選）
tasks.md任務清單 — 具體的實作步驟，勾選式

5-3. Delta Specs（差異規格）

不是重寫整份規格，而是只描述變化的部分：

區段意思
ADDED新增的需求
MODIFIED修改的需求（會標示舊版內容以供對照）
REMOVED移除/廢棄的需求

5-4. Archive（歸檔）

完成實作後，歸檔會做三件事：

  1. 將 delta specs 合併回主規格（specs/ 資料夾）
  2. 將整個 change 資料夾移到 changes/archive/YYYY-MM-DD-名稱/
  3. 保留完整歷史紀錄，供日後查閱

5-5. Schemas（工作流程模板）

定義有哪些文件、文件之間的依賴關係。預設使用 spec-driven 模板，你也可以自訂。依賴是「啟用器」而非「門檻」—— 顯示什麼可以做，而不強制順序。

六、工作流程模式

模式 A：核心模式（Core Profile，預設）

只用 3 個指令，簡單快速：

/opsx:propose 提案 + 規劃 → /opsx:apply 實作程式碼 → /opsx:archive 歸檔完成

模式 B：擴展模式（Expanded Profile）

更細緻的控制，適合大型功能：

/opsx:new → continue / ff → /opsx:apply → /opsx:verify → /opsx:archive

切換到擴展模式：

openspec config profile expanded
openspec update

七、所有指令詳解

7-1. Slash Commands（在 AI 工具對話中使用）

指令說明範例
/opsx:propose <名稱> 建立變更 + 一次產生所有規劃文件 /opsx:propose add-dark-mode
/opsx:explore <主題> 探索模式，調查問題、釐清需求（不產生文件） /opsx:explore performance-issues
/opsx:apply [名稱] 按照 tasks.md 開始實作程式碼 /opsx:apply add-dark-mode
/opsx:archive [名稱] 歸檔完成的變更，合併規格 /opsx:archive add-dark-mode
/opsx:new <名稱> 只建立變更的骨架（不產生文件內容） /opsx:new fix-login-bug
/opsx:continue [名稱] 逐步產生下一個文件（一次一個，可檢查） /opsx:continue fix-login-bug
/opsx:ff [名稱] 快速一次產生所有規劃文件（Fast-Forward） /opsx:ff fix-login-bug
/opsx:verify [名稱] 驗證實作是否符合規格（完整性、正確性、一致性） /opsx:verify fix-login-bug
/opsx:sync [名稱] 將 delta specs 合併到主規格（不歸檔） /opsx:sync fix-login-bug
/opsx:bulk-archive 批次歸檔多個完成的變更 /opsx:bulk-archive
/opsx:onboard 互動式新手教學（帶你跑一次完整流程） /opsx:onboard

不同工具的語法差異

工具格式範例
Claude Code冒號 :/opsx:propose
Cursor, Windsurf, Copilot連字號 -/opsx-propose
Trae完整名稱/openspec-propose

7-2. CLI 終端指令（在命令列中使用）

指令說明
openspec init初始化專案
openspec update升級後更新指令檔
openspec list --specs列出所有規格
openspec list --changes列出所有變更
openspec show <名稱>顯示變更或規格的詳細內容
openspec view開啟互動式儀表板
openspec validate --all檢查所有文件結構是否正確
openspec status --change <名稱>顯示文件完成進度
openspec archive <名稱>從終端歸檔變更
openspec config profile <模式>切換工作流程模式
openspec schemas列出可用的工作流程模板
openspec config list查看目前所有設定
openspec config set <key> <value>修改設定值
openspec schema fork <來源> <名稱>複製工作流程模板來自訂
openspec completion install <shell>安裝 shell 自動補全

八、實際操作範例（完整走一次）

以「為專案新增深色模式」為例：

步驟 1：提案

在 Claude Code 中輸入：

/opsx:propose add-dark-mode

OpenSpec 會自動建立以下結構：

openspec/changes/add-dark-mode/
├── .openspec.yaml    ← 中繼資料
├── proposal.md      ← 為什麼要做深色模式
├── specs/            ← 需要新增/修改哪些行為
├── design.md        ← 技術設計方案
└── tasks.md         ← 實作步驟清單

步驟 2：實作

/opsx:apply add-dark-mode

AI 會讀取 tasks.md 中的任務清單，逐一實作程式碼，完成一項就勾一項。

中斷後可繼續

如果中途中斷，再次執行 /opsx:apply add-dark-mode 會從上次中斷處繼續，不會重頭開始。

步驟 3：驗證（擴展模式才有）

/opsx:verify add-dark-mode

檢查三個面向：

  • 完整性 — 所有任務都完成了嗎？所有需求都滿足了嗎？
  • 正確性 — 實作是否符合設計意圖？
  • 一致性 — 程式碼是否反映了設計文件的架構？

步驟 4：歸檔

/opsx:archive add-dark-mode

這會：

  1. 將 delta specs 合併到主規格（specs/ 資料夾）
  2. 將整個 change 移到 changes/archive/2026-03-02-add-dark-mode/
  3. 保留完整歷史紀錄

九、常見工作流程模式

模式 1：快速功能（需求明確時）

/opsx:propose → /opsx:apply → /opsx:archive

適合：小到中型功能、Bug 修復、範圍已確定的工作。

模式 2：探索式開發（不確定怎麼做時）

/opsx:explore → /opsx:propose → /opsx:apply → /opsx:archive

適合：效能優化、除錯、架構決策等需要先調查的工作。

模式 3：細緻控制（大型功能時）

/opsx:new big-feature
/opsx:continue big-feature    ← 產生 proposal.md，你檢查
/opsx:continue big-feature    ← 產生 specs/，你檢查
/opsx:continue big-feature    ← 產生 design.md，你檢查
/opsx:continue big-feature    ← 產生 tasks.md，你檢查
/opsx:apply big-feature       ← 開始實作
/opsx:verify big-feature      ← 驗證
/opsx:archive big-feature     ← 歸檔

模式 4：平行開發（同時做多個功能）

每個功能各自獨立，隨時可以切換：

/opsx:apply feature-a    ← 做 A
/opsx:apply feature-b    ← 切去做 B
/opsx:apply feature-a    ← 切回來繼續 A
如何選擇 /opsx:ff 還是 /opsx:continue？
  • ff（快轉）— 需求明確、趕時間時用，一次生成所有文件
  • continue（逐步）— 需要探索、想每步檢查時用，一次生成一個文件

十、客製化設定

10-1. 專案設定檔 (openspec/config.yaml)

可以加入專案上下文，讓 AI 更理解你的技術棧：

# openspec/config.yaml

defaults:
  schema: spec-driven

context: |
  Tech stack: TypeScript, React, Node.js, PostgreSQL
  API convention: RESTful, snake_case
  Testing: Jest + React Testing Library

rules:
  - artifact: proposal
    rule: "必須包含 rollback 計畫"
  - artifact: tasks
    rule: "每個任務必須包含測試步驟"

10-2. 自訂工作流程模板 (Schema)

# 複製現有模板來修改
openspec schema fork spec-driven my-workflow

# 或建立全新模板
openspec schema init rapid --artifacts "proposal,tasks" --default

# 驗證模板語法
openspec schema validate my-workflow

10-3. Schema 解析順序

OpenSpec 按以下順序決定使用哪個 schema：

  1. CLI 旗標（--schema <name>）
  2. Change 的中繼資料（.openspec.yaml）
  3. 專案設定（config.yaml）
  4. 內建預設（spec-driven）

十一、升級 OpenSpec

# 升級 CLI 到最新版
npm install -g @fission-ai/openspec@latest

# 更新專案中的指令檔（在專案資料夾內執行）
cd 你的專案
openspec update

十二、新手建議的第一步

如果你是第一次使用，最推薦使用互動式新手教學：

/opsx:onboard

這個指令會：

  1. 掃描你的程式碼庫，了解專案結構
  2. 帶你建立一個真實的小型變更
  3. 走過完整的 提案 → 實作 → 歸檔 流程
  4. 每一步都有說明與引導
最佳實踐建議
  • 每個 Change 保持聚焦，代表一個單一邏輯單元
  • 不確定時先用 /opsx:explore 探索
  • 歸檔前用 /opsx:verify 驗證
  • 使用描述性名稱（如 fix-login-timeout 而非 fix1）
  • 推薦使用高推理能力的模型（如 Opus、GPT 5.2）

十三、隱私與遙測

OpenSpec 預設會收集匿名使用資料（只有指令名稱和版本號，不含路徑、內容或個人資訊）。在 CI 環境中會自動關閉。

手動關閉方式：

# 二選一
export OPENSPEC_TELEMETRY=0
export DO_NOT_TRACK=1

# 或用 OpenSpec 內建設定
openspec config set telemetry.enabled false
